## 阶段一：系统部署与内部测试（第1周）

### 目标
- 系统在服务器上正常运行
- 核心团队成员完成培训，能独立操作系统
- 用历史数据验证系统准确性

### 任务清单

| 序号 | 任务 | 负责人 | 验收标准 | 预计耗时 |
|------|------|--------|----------|----------|
| 1.1 | 服务器环境部署（Python 3.10+, 依赖安装） | IT | uvicorn 正常启动，/healthz 返回 OK | 0.5 天 |
| 1.2 | 导入历史订单数据（如有） | 运营 | 数据成功导入 data/nanning/，可在复盘页面查看 | 1 天 |
| 1.3 | 运行数据生成脚本 | IT | 7 个 CSV 文件生成成功 | 0.5 天 |
| 1.4 | 训练 AI 定价模型 | IT | `pricing_model.json` 生成，模型信息 API 返回正常 | 0.5 天 |
| 1.5 | 核心团队培训（销售主管+运营经理+总经理） | 项目组 | 每人独立完成一次完整报价流程 | 1 天 |
| 1.6 | 用历史订单验证系统准确性 | 运营 | 回测显示 AI 定价亏损率 ≤ 旧定价 | 0.5 天 |
| 1.7 | 配置初始成本参数（对齐实际成本） | 运营+财务 | 参数经财务确认无误 | 0.5 天 |
| 1.8 | 准备销售培训材料和 FAQ | 运营 | 培训材料打印/发送完毕 | 0.5 天 |

### 里程碑验收

- [ ] 系统在生产服务器正常启动，API 和前端均可正常访问
- [ ] 导入历史数据，回测结果显示 AI 定价不差于人工定价
- [ ] 核心团队（≥3人）完成操作培训，可独立使用
- [ ] 销售培训材料和 FAQ 准备就绪

## 阶段二：青秀区试点运行（第2-3周）

### 目标
- 在青秀区真实业务中验证系统效果
- 收集一线销售反馈，调整规则参数
- 确保销售团队适应新定价模式

### 试点范围

| 维度 | 范围 |
|------|------|
| 区域 | 仅青秀区（业务量最大、价格敏感度最低） |
| 服务 | 全部 4 种服务类型 |
| 人员 | 青秀区销售团队（约 3-5 人） |
| 模式 | AI 定价为主，可手动申请优惠 |

### 任务清单

| 序号 | 任务 | 负责人 | 验收标准 | 预计耗时 |
|------|------|--------|----------|----------|
| 2.1 | 青秀区销售团队培训（小程序使用+议价规范） | 销售主管 | 每人完成 3 次模拟报价，正确率 100% | 第1天 |
| 2.2 | 正式切换青秀区报价为系统报价 | 销售主管 | 青秀区所有新单使用系统报价 | 第2天 |
| 2.3 | 每日跟踪：记录系统报价 vs 实际成交价偏差 | 运营 | 每日记录偏差原因（议价/加项/工时偏差等） | 每天 |
| 2.4 | 第一周回顾：分析报价准确性和亏损率 | 运营+项目组 | 出具第1周试点报告 | 第7天 |
| 2.5 | 根据反馈调整成本参数或规则 | 运营+项目组 | 调整后的参数经验证无误 | 第7-8天 |
| 2.6 | 第二周继续试点，验证调整效果 | 销售主管 | 第二周亏损率 ≤ 第一周 | 第8-14天 |
| 2.7 | 试点总结报告 | 运营+项目组 | 报告包含：关键指标、问题清单、改进措施 | 第14天 |

### 试点期重点观察指标

| 指标 | 目标值 | 警戒线 |
|------|--------|--------|
| 系统报价采纳率 | ≥ 80% | < 60% |
| 报价后成交率 | ≥ 60% | < 40% |
| 订单亏损率 | ≤ 5% | > 10% |
| 平均毛利率 | ≥ 20% | < 15% |
| 销售满意度（问卷） | ≥ 4/5 分 | < 3/5 分 |

### 试点退出条件

- [ ] 系统报价采纳率 ≥ 80%
- [ ] 订单亏损率 ≤ 5%
- [ ] 核心参数已根据试点反馈调整到位
- [ ] 销售团队反馈积极，无重大操作障碍
- [ ] 试点报告通过管理层审核

## 阶段三：全公司推广上线（第4周）

### 目标
- 7 城区全部切换到系统定价
- 全量订单使用 AI 定价
- 完成全员培训

### 任务清单

| 序号 | 任务 | 负责人 | 验收标准 | 预计耗时 |
|------|------|--------|----------|----------|
| 3.1 | 全员培训（分 2 批次：销售+运营） | 项目组 | 全员考核通过（正确完成报价+议价流程） | 1 天 |
| 3.2 | 按城区分批次切换（先中端城区，后远郊） | 运营 | 每天切换 2-3 个城区 | 3 天 |
| 3.3 | 上线首周每日监控数据 | 运营 | 每日出具 KPI 快报 | 每天 |
| 3.4 | 上线周报 | 运营 | 包含全量 KPI、问题汇总、改进建议 | 第7天 |
| 3.5 | 管理层汇报 | 项目组 | 演示系统运行效果，汇报关键指标 | 第7天 |

### 切换顺序

```
第1天：江南、西乡塘（订单量中等，风险可控）
第2天：兴宁、良庆（五象）
第3天：邕宁、武鸣（远郊，需确认交通加价是否足够）
     （青秀已在阶段二完成切换）
```

### 推广验收标准

- [ ] 7 城区全部使用系统报价
- [ ] 全公司销售/运营人员完成培训并通过考核
- [ ] 上线首周亏损率 ≤ 5%
- [ ] 无因系统定价导致的重大客户投诉
- [ ] 管理层确认项目达到预期效果

## 阶段四：系统迭代优化（长期）

### 月度运维

| 项目 | 频率 | 负责人 |
|------|------|--------|
| 成本参数复核（师傅薪资、交通费变动等） | 每月 | 运营+财务 |
| 订单复盘 + 亏损分析 | 每月 | 运营 |
| 模型效果评估（AI vs 规则） | 每月 | IT |
| 竞品价格监测更新 | 每月 | 运营 |

### 季度优化

| 项目 | 说明 |
|------|------|
| 模型重训练 | 用最新订单数据重新训练 XGBoost 模型 |
| 规则优化 | 根据季度数据调整季节系数、区域加价 |
| 功能迭代 | 根据用户反馈增加新功能 |
| 竞品分析报告 | 深度分析南宁市场格局变化 |

### 年度规划

| 阶段 | 目标 |
|------|------|
| Q1-Q2（当前） | 系统稳定运行，AI 定价全面覆盖 |
| Q3 | 接入实时竞品价格监控，增强市场锚定层 |
| Q4 | 探索强化学习模型，实现更智能的动态调价 |
| 次年 | 扩展至南宁周边城市（柳州、桂林），验证模型可迁移性 |

## 风险与预案

| 风险 | 概率 | 影响 | 预案 |
|------|------|------|------|
| 销售不使用系统报价 | 中 | 高 | 强制要求所有报价通过系统生成，纳入绩效考核 |
| AI 定价偏高导致丢单 | 中 | 中 | 随时可切换为纯规则模式（更保守的定价） |
| 成本参数不准确 | 低 | 高 | 阶段一由财务逐项确认，试点期修正 |
| 旺季系统压力大 | 低 | 低 | FastAPI 异步架构可承受，必要时增加 worker |
| 小程序体验差 | 中 | 中 | 同时提供 Web SPA 作为备选 |
